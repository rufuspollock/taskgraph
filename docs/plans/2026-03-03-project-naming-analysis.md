# Project naming analysis

## Purpose

Evaluate whether "taskgraph" is the right name for the project, and whether "task" or "issue" is the right term for the core item.

## Task vs Issue

"Task" is the better term. The vision essay is about finding actionable work — leaf tasks, concrete things to do next. "Issue" carries baggage from bug trackers (GitHub Issues, Jira) and implies problems to fix. This project is broader: choosing what to work on next across all work, not triaging problems.

## Name candidates

| Name | Domain | Notes |
|------|--------|-------|
| **taskgraph** | taskgraph.dev (owned) | Clear, accurate, googleable. Honest. |
| **tasktree** | tasktree.app (taken) | More intuitive visual metaphor — the vision essay uses tree-like descent. Slightly warmer than "graph". |
| **nexttask** | nexttask.dev | Captures the purpose — "what should I do next?" |
| **taskdrill** | taskdrill.dev | Captures the drilling-down motion central to the vision. |
| **leafwork** | leafwork.dev | Highlights the key insight: reaching the leaf node. |

"taskgraph" is solid: clear, searchable, no cleverness to explain. The only thing it slightly misses is the dynamism of the descent/navigation, but that's what taglines are for.

The strongest alternative is "tasktree" — same clarity but evokes hierarchical navigation better than "graph" does for most people.

## Conclusion

Sticking with **taskgraph** (taskgraph.dev). The best alternative would have been "tasktree" but it's taken and active (tasktree.app), so not worth the confusion. "taskgraph" is already owned, accurate, and ages well.
